概述

系统整体领域分为三个部分

  1. Cloud 云端
  2. Edge 边端
  3. Device 设备端

其中系统架构主要集中在两个部分:

  1. 云端部分
    • CloudHub: CloudHub 是一个 Web Socket 服务端，负责监听云端的变化, 缓存并发送消息到 EdgeHub。
    • EdgeController: EdgeController 是一个扩展的 Kubernetes 控制器，管理边缘节点和 Pods 的元数据确保数据能够传递到指定的边缘节点。
    • DeviceController: DeviceController 是一个扩展的 Kubernetes 控制器，管理边缘设备，确保设备信息、设备状态的云边同步。
  2. 边端部分
    • EdgeHub: EdgeHub 是一个 Web Socket 客户端，负责与边缘计算的云服务（例如 KubeEdge 架构图中的 Edge Controller）交互，包括同步云端资源更新、报告边缘主机和设备状态变化到云端等功能。
    • Edged: Edged 是运行在边缘节点的代理，用于管理容器化的应用程序。
    • EventBus: EventBus 是一个与 MQTT 服务器（mosquitto）交互的 MQTT 客户端，为其他组件提供订阅和发布功能。
    • ServiceBus: ServiceBus是一个运行在边缘的HTTP客户端，接受来自云上服务的请求，与运行在边缘端的HTTP服务器交互，提供了云上服务通过HTTP协议访问边缘端HTTP服务器的能力。
    • DeviceTwin: DeviceTwin 负责存储设备状态并将设备状态同步到云，它还为应用程序提供查询接口。
    • MetaManager: MetaManager 是消息处理器，位于 Edged 和 Edgehub 之间，它负责向轻量级数据库（SQLite）存储/检索元数据。

云端部分

CloudHub

  • 负责与EdgeHub通信
  • 支持 websocket 和 QUIC 协议

CloudHub 主要功能:

  • 获取 message context 并创建 ChannelQ events
    • message context 对象存储在 channelQ. 对于所有创建的 nodeID channel, message context 转换为 event 对象然后传送到chennel.
  • 创建 http-over-websocket 连接
    • TLS证书通过上下文对象中提供的路径加载
    • HTTP服务器以TLS配置启动
  • 服务 websocket连接
    • 然后将HTTP连接升级到WebSocket连接接收Conn对象
    • ServeConn函数，用于所有传入连接
  • 从边缘读取消息
  • 往边缘写消息
    • First all event objects are received for the given nodeID
    • The existence of same request and the liveness of the node is checked
    • The event object is converted to message structure
    • Write deadline is set. Then the message is passed to the websocket connection
  • 推送消息到Controller
    • 每次请求都通过websocket连接发送到controller,认的消息附带timestamp, clientID 和 event type
    • 如果节点断开连接，则抛出错误，并且将描述节点故障的事件推送到controller。

Edge Controller

边端部分

EdgeHub

  • EdgeHub 负责与CloudHub交互
  • 使用websocket 或 QUIC 协议连接到 CloudHub
  • 支持同步云端资源的更新
  • 报告边端主机和设备的状态变更
  • 将从云接收的消息转发到边缘模块, 反之亦然

主要功能:

  • keep alive

    • 通过 keep-alive 或者 heartbeat 消息与 CloudHub 之间保活

  • publish client info

  • route to cloud

  • route to edge

MetaManager

MetaManager 在 edged 和 edgehub 处理消息. 也从 SQLite 中存储和检索消息.

Metamanager 接收以下几种类型消息:

  • Insert Operation

    • 当创建新的对象时, 通过云端接收 Insert 消息. 例如通过云端创建/部署新的Pod.
    • edgehub 通过云端接收 insert operation. 然后发送消息到 metamanager 并存储在本地的数据库.
    • metamanager 发送匿名消息到 edged, edged 处理 insert 请求, 例如启动 pod 并填充响应.
    • metamanager 检查消息, 提取响应并发回 edged 和云端.

  • Update Operation

    • Update operation 会在云端、边端发生.
    • update 消息流类似insert.
    • 此外, MetadataManager 检查本地资源是否更改. 只有当资源发生更改才会更新本地资源并发送响应到边端和云端, 这里的检查是增量的

  • Delete Operation

    • delete 操作发生在云端. 例如在云端删除一个Pod.

  • Query Operation

    • Query operation 可以在边端查询本地元数据, 也可以通过 metammanager 查询云端的资源 (如 configmap/secret 等).
    • metadatamanager 返回响应给edged, 资源消息格式为 resKey/resType/resId

  • Response Operation

    • Response operation 是在云端和边端执行任何操作后的返回. 上面其他的操作描述了 Response 在云端和边端流动.

  • NodeConnection Operation

    • 从EdgeHub接收NodeConnection操作消息，以提供云端连接状态的信息。
    • Metamanager在内存中跟踪此状态，并在某些操作中使用它，如通过云端查询。

  • MetaSync Operation

    • Metamanager 周期性地发送 MetAsync 操作消息以同步在边缘节点上运行的POD的状态。
    • 同步间隔可在Conf/Edge.yaml中配置（默认为60秒）。

DeviceTwin

  • 存储设备状态
  • 处理设备属性
  • 处理设备 twin 操作
  • 在边缘设备和边端节点之间创建成员身份
  • 将设备的状态与云端同步, 并同步边端和云端的设备 twin 信息
  • 为应用程序提供查询接口
  • DeviceTwin 由4个子模块组成
    • namely membership
    • communication
    • device
    • device twin

DeviceTwin Controller

执行以下操作:

  • Sync metadata to/from db ( Sqlite )
  • Register and Start Sub Modules
  • Distribute message to Sub Modules
  • Health Check

Modules

Membership Module
  • 通过云端添加边缘设备的membership
  • 将新添加的设备绑定到边缘节点, 并在边缘节点和边缘设备之间创建membership

该模块主要执行的功能:

  1. 初始化 action callback map, 是一个 map[string]Callback 包括要执行的回调函数
  2. 接收 message 发送到 membership module
  3. 对于每个消息，读取消息相关的 action, 并调用相应的函数
  4. 从 heartbeat channel 接收 heartbeat 消息并发送 heartbeat 到 controller

下面是 membership moudle 可以执行的回调:

  • dealMembershipGet

    • dealMembershipGet() 从缓存中获取与特定边缘节点关联的设备信息
    • eventbus 首先从它订阅的topic (membership-get topic) 接收一条消息
    • 该消息到达 devicetwin controller, 进一步将消息发送到 membership module
    • membership module 从缓存 (context) 中获取与边缘节点关联的设备, 并将信息发送到通信模块
    • 它还处理在执行上述过程中可能发生的错误, 并将错误发送到通信模块而不是设备详细信息
    • 通信模块将信息发送到 eventbus, eventbus 进一步发布到指定的 MQTT topic (get membership result topic)

  • dealMembershipUpdated

    • dealmembershipddated() 更新节点的成员资格详细信息, 它负责新设备在 edge group 的添加和删除, 如果有更新, 则更新设备详细信息
    • edgehub 接收 CloudHub 发来的 membership 更新消息, 并将消息转发给 devicetwin controller
    • membership moudle 添加新的设备, 删除最近删除的设备, 并更新数据库和缓存中已经存在的设备
    • 更新设备的详细信息后, 将向 device twin 的 communication module 发送一条消息，该模块将消息发送到 eventbus 模块，以便在相应的 MQTT 主题上发布。

  • dealMembershipDetail

    • dealMembershipDetail() 提供边缘节点 membership details, 在删除最近删除设备的 membership detail 后, 提供关联边缘节点的设备信息
    • eventbus module 接收订阅的topic发送的消息, 将该消息转发到 devicetwin controller, devicetwin controller 进一步转发到 membership module
    • membership module 添加消息中涉及到设备, 删除缓存中不存在的设备
    • 更新设备的detail后, 想 devicetwin 的 combination module 发送一条消息
Twin Module

twin 模块的主要职责是处理所有与 device twin 相关的操作. 它可以执行 device twin update, get 以及 sync-to-cloud.

Communication Module

communication 模块的主要职责是确保 device twin 与其他组件之间的通信功能.

Device Module

device 模块的主要职责是执行与设备相关的操作，例如处理设备状态更新和设备属性更新。

EventBus

  • EventBus 作为 在 MQTT 相关 topic 上发送/接收消息的接口.
  • 支持3种模式：
    • InternalMQTTMode
    • ExternalMQTTMode
    • BothMQTTMode
  • EventBus订阅以下topic:
    • $hw/events/upload/#
    • SYS/dis/upload_records
    • SYS/dis/upload_records/+
    • $hw/event/node/+/membership/get
    • $hw/event/node/+/membership/get/+
    • $hw/events/device/+/state/update
    • $hw/events/device/+/state/update/+
    • $hw/event/device/+/twin/+

Note: topic 通配符

  • #: It must be the last character in the topic, and matches the current tree and all subtrees.
  • +: It matches exactly one item in the topic tree.
